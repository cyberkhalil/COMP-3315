Python Programming Project Assignment

Network Socket Programming | COMP‑3315 | Spring 2025

1. Project Objective

The primary objective of this practical programming project is to provide students with hands-on, experiential learning in network socket programming using Python. By designing and deploying a working client-server architecture, students will bridge the gap between theoretical networking concepts (such as transport-layer protocols, state management, multiplexing, and byte-stream handling) and real-world application development. Students are expected to implement robust network configurations using either TCP or UDP, depending on their chosen project application.

2. Project Options

Students may work individually or in designated groups (subject to instructor approval). You must select ONE of the following project specifications to implement:

Option 1: Multi-User Chat Application (TCP-Based)

Description: Develop a centralized chat server capable of managing multiple concurrent client connections. The server must handle incoming messages from various users and broadcast them dynamically to all other active clients in real-time.

Core Protocols & Libraries: TCP Sockets, using standard built-in socket and threading modules.

Concurrency: The server must employ multi-threading to handle asynchronous client inputs seamlessly without blocking the main execution loop.

Identity Management: Clients must prompt users for a unique nickname or username upon initial connection, which will preface all broadcasted messages.

TCP · threading
Option 2: Secure File Transfer Protocol - FTP (TCP-Based)

Description: Build a dedicated file transfer system engineered for reliable uploading and downloading of diverse file types over a local network infrastructure.

Reliable Transfer: Must leverage TCP sockets to guarantee packet delivery, strict packet ordering, and data integrity.

Command Structure: The client interface must support standard actions: listing available files on the server directory, uploading a local file, and downloading a remote file.

Data Handling: Correctly manage binary streams to facilitate the transfer of non-text assets (such as images, PDFs, or executables) without corruption, accounting for progressive buffering.

TCP · binary streaming
Option 3: Mini Web Browser & HTTP Web Server (TCP-Based)

Description: Implement a low-level HTTP client (browser simulation) that constructs and transmits raw HTTP requests to a target web server and parses the returning HTML payload. Alternatively, construct a lightweight HTTP web server capable of serving static HTML assets to standard commercial browsers.

Protocol Compliance: Adhere strictly to the basic syntax of the HTTP/1.1 protocol.

Request/Response Processing: Handle standard GET requests, structural headers, and return accurate status codes (e.g., 200 OK, 404 Not Found, 400 Bad Request).

HTTP/1.1 · raw sockets
Option 4: Remote Command Execution Tool / RAT (TCP-Based)

Description: Design an administrative tool allowing a remote client to securely send operating system terminal commands to a target server. The server executes these commands locally and pipes the standard output/error back to the client console.

Execution Engine: Utilize Python's built-in subprocess module to spawn shell contexts safely.

Error Handling: Build resilient handling mechanisms to capture and forward runtime terminal errors without crashing the backend socket server.

subprocess · error resilience

3. General Technical Requirements

  • Development Environment: All code must be written natively in Python 3.x using standard built-in networking suites (socket, threading). Third-party high-level network frameworks are strictly prohibited.
  • Architectural Separation: Maintain an absolute, decoupled separation between client-side logic (client.py) and server-side infrastructure (server.py).
  • Robust Exception Handling: Codebases must defensively anticipate network irregularities. Incorporate explicit try-except structures to manage unexpected client disconnections, connection timeouts, and port binding conflicts.
  • User Interface: A clean, interactive Command Line Interface (CLI) is the baseline standard. Integrating a Graphical User Interface (GUI) via libraries like tkinter is entirely optional, but will earn bonus points.

4. Deliverables

All student groups are required to submit a single, compressed archive file (.zip) containing the following mandatory components:

  • Source Code Directory: Fully functioning and well-documented server.py and client.py scripts.
  • Project Report: A brief report (ReadMe.md or PDF) detailing configuration instructions, system screenshots demonstrating active communication, and an analytical paragraph justifying your transport layer protocol selection (TCP vs UDP).
  • Video Demonstration (Maximum 10 Minutes): A comprehensive screen-recorded video featuring clear vocal commentary. The video must cover:
    • A high-level structural walkthrough of your codebase, highlighting how network sockets are initialized and managed.
    • A live, unedited execution demonstration of the application proving real-time interaction, functional data exchange, and graceful handling of edge cases or runtime errors.

5. Evaluation Criteria

Assessment Component Weight Key Metrics
Functionality & Live Demo 40% Seamless end-to-end operational capability during the live video execution; absence of fatal runtime exceptions; compliance with basic option goals.
Code Architecture & Handling 30% Appropriate utilization of socket methods, robust multi-threading or sequential handling loops, clean code layouts, and comprehensive exception parsing.
Video Explanation & Clarity 15% Articulate verbal delivery, strong conceptual grasp of network socket lifecycles, clear code navigation, and strict adherence to the 10-minute maximum limit.
Documentation & Discussion 15% Precision of configuration rules in the report file, visual evidence via screenshots, and conceptual readiness displayed during in-person discussions.
